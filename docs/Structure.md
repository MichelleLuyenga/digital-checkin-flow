### Key structural decisions explained

**1. PMS Adapters as a pluggable module**  
`src/lib/pms-adapters/interface.ts` defines a strict TypeScript interface (`IPMSAdapter`) for all PMS operations (update guest status, get rooms, assign room, issue key).  
Concrete adapters live in the same folder: `mock-adapter.ts` returns predefined data and logs calls; `opera-adapter.ts` would use a SOAP client; `mews-adapter.ts` uses `fetch`. The API routes never import a specific adapter directly – they receive one via dependency injection (or environment variable) at build time. This is the heart of making the transition from polished UI to real integration.

**2. Supabase as the source of truth**  
All user-facing state (check-in progress, identity status, preferences) lives in Supabase tables. API routes write to Supabase first, then call the PMS adapter. The frontend subscribes to real-time changes (e.g., “room assignment confirmed”) via Supabase Realtime hooks.  
The `supabase/` folder contains migrations to create the tables (`checkin_sessions`, `reservations`, `identity_verifications`, `preferences`, `qr_tokens`) and the Row-Level Security (RLS) policies that tie a guest to their own data.

**3. The `c/[token]` dynamic route**  
This is the entire pre-arrival check-in flow. The page server component validates the token and passes the reservation to a client component that orchestrates the multi-step UI. The steps are Identity → Preferences → QR Code.

**4. API routes as HTNG-compliant endpoints**  
Each route under `api/checkin/` corresponds to a logical step in the HTNG Guest Self-Service specification. They’re designed to be idempotent and return structured error codes that the frontend can interpret (e.g., “room not available” triggers a preference re-selection).  
The `/api/qr/validate` endpoint is separate – it’s what the door lock or lobby kiosk calls. It verifies the JWT/QR token and returns the guest’s room number and validity period.

**5. Queue for eventual consistency**  
`lib/queue/pms-retry.ts` is a placeholder for a real queue (BullMQ + Redis). When a PMS update fails, the API route enqueues a retry job and returns a “pending” status. The frontend shows “Syncing with hotel system…” until the retry succeeds and Supabase is updated (which triggers a realtime event to the UI).

**6. PWA shell & service worker**  
`public/manifest.json` defines how the app appears when installed (full-screen, orientation, theme colour). The service worker is auto-generated by `next-pwa`, which also pre-caches the app shell for instant offline loading. The rest of the app requires network, but we show graceful offline fallback messages.

This structure ensures that the “polished UI” and the “messy PMS reality” are cleanly separated, yet tightly connected through well-defined interfaces. When it’s time to integrate with a real Opera PMS, you only need to fill in `opera-adapter.ts` and swap the import – the entire customer-facing experience remains unchanged.
